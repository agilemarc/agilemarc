📖 O Que São User Stories?

A base de tudo no desenvolvimento ágil

Definição Simples

User Stories são descrições curtas e simples de uma funcionalidade contada da perspetiva da pessoa que a deseja. Não são requisitos técnicos detalhados - são conversas sobre valor que guiam o desenvolvimento.

Princípio Fundamental

"As melhores User Stories nascem de conversas, não de documentos."

✨ Características Essenciais

  • Simples: Linguagem não-técnica que todos entendem
  • Centradas no utilizador: Foco em quem vai usar
  • Orientadas ao valor: Sempre com propósito claro
  • Negociáveis: Ponto de partida para discussão
  • Testáveis: Critérios claros de sucesso

💡 Porquê User Stories?

  • Mantém foco no cliente, não na tecnologia
  • Promove colaboração Dev + PO
  • Flexibilidade para mudanças
  • Entrega incremental de valor
  • Reduz desperdício e retrabalho

✍️ Formato e Anatomia

Como escrever User Stories perfeitas

Formato Clássico

Como [tipo de utilizador]
Quero [funcionalidade]
Para que [benefício/valor]

História Bem Escrita - Contexto Bancário

Como gestor de conta empresarial
Quero aprovar pagamentos em lote via mobile
Para que possa gerir aprovações mesmo fora do escritório

✨ Porque funciona:
  • Persona específica e clara
  • Necessidade bem definida
  • Valor de negócio explícito
  • Abre espaço para discussão técnica

História Mal Escrita

"Implementar sistema de aprovações"

❌ Problemas:
  • Quem precisa? Não sabemos
  • Que tipo de aprovações?
  • Qual o valor de negócio?
  • Mobile? Desktop? API?

🎯 Os 3 C's da User Story

📋 Card (Cartão)

A descrição escrita da história no formato padrão

Como [persona] Quero [ação] Para [valor]

💬 Conversation (Conversa)

Discussões para clarificar detalhes

  • Que tecnologia usar?
  • Quais as regras de negócio?
  • Como lidar com erros?
  • Qual a performance esperada?

✅ Confirmation (Confirmação)

Critérios de aceitação claros

DADO QUE [contexto] QUANDO [ação] ENTÃO [resultado]

📊 Aplicando INVEST na Prática

Story Original (Falha INVEST)

"Implementar sistema de autenticação completo com todas as funcionalidades"

  • ❌ Independent: Bloqueia tudo
  • ❌ Negotiable: "Completo" é vago
  • ❓ Valuable: Valor não claro
  • ❌ Estimable: Grande demais
  • ❌ Small: Meses de trabalho
  • ❌ Testable: Sem critérios

Stories Refinadas (INVEST Compliant)

Story 1: "Como utilizador, quero fazer login com email/password para aceder à minha conta"

Story 2: "Como utilizador, quero recuperar password esquecida para retomar acesso"

Story 3: "Como utilizador, quero autenticação 2FA para maior segurança"

  • ✅ Cada uma é independente
  • ✅ Scope negociável
  • ✅ Valor claro para utilizador
  • ✅ 2-3 dias cada
  • ✅ Critérios testáveis

🔄 Processo de Refinamento

  1. Identificar dependências: Separar partes independentes
  2. Clarificar valor: Adicionar o "para que"
  3. Dividir em chunks: Max 5 dias de trabalho
  4. Definir critérios: Given-When-Then claros
  5. Validar com equipa: Todos conseguem estimar?

🗺️ Story Mapping

A técnica que revoluciona o planeamento de produtos

O que é Story Mapping?

Story Mapping é uma técnica visual para organizar user stories numa narrativa que mostra a jornada completa do utilizador. É como um mapa do tesouro para o produto!

🔄 Eixo Horizontal

Sequência temporal da jornada do utilizador

Da esquerda para direita: início ao fim

📊 Eixo Vertical

Prioridade e sofisticação das funcionalidades

De cima para baixo: essencial ao nice-to-have

🏗️ Como Construir um Story Map

1. Backbone (Atividades)

Identificar grandes blocos de atividade do utilizador

Exemplo E-commerce: Descobrir → Avaliar → Comprar → Receber → Suporte

2. Walking Skeleton (Tarefas)

Decompor cada atividade em tarefas específicas

"Comprar": Ver carrinho → Inserir dados → Pagar → Confirmar

3. User Stories

Adicionar stories específicas sob cada tarefa

"Pagar": Cartão crédito → PayPal → MBWay → Crypto

4. Releases

Dividir verticalmente em entregas incrementais

MVP: Funcionalidades mínimas
R2: Melhorias UX
R3: Features avançadas

🎯 Story Mapping Avançado - Gestão de Dependências

⚡ Dependências Técnicas

Sistema Autenticação → Bloqueia:

  • Subscrições (precisa user ID)
  • Histórico (precisa sessão)
  • Personalização (precisa perfil)
🚨 Mitigação: Começar na Sprint 1

💼 Dependências Negócio

Compliance/Legal → Necessário para:

  • KYC/AML procedures
  • GDPR compliance
  • Termos de serviço
⚠️ Mitigação: Envolver legal desde início

🌐 Dependências Externas

APIs Terceiros → Setup necessário:

  • Gateway pagamentos (5 dias)
  • Provider SMS (2 dias)
  • Serviço KYC (10 dias)
💡 Mitigação: Iniciar contratos Sprint 0

✂️ Técnicas de Splitting

📱 Por Interface

Web → Mobile → API → Admin

Original: Sistema alertas
Split: Email | SMS | Push | In-app

📊 Por Workflow

Criar → Ler → Atualizar → Apagar

Original: Gestão beneficiários
Split: Adicionar | Listar | Editar | Remover

⚡ Por Performance

Funcional → Rápido → Otimizado

Original: Dashboard real-time
Split: 5min cache | 30seg | Real-time

🎯 Por Regras

Simples → Complexo → Edge cases

Original: Aprovação crédito
Split: <10K auto | <50K manual | >50K comité

🔢 Por Dados

Tipo A → Tipo B → Tipo C

Original: Tipos pagamento
Split: Nacional | SEPA | Internacional

👥 Por Persona

Cliente → Gestor → Admin

Original: Dashboard completo
Split: Vista cliente | Vista gestor | Vista compliance

🔬 Spikes - Investigação com Propósito

O que é um Spike?

Spike é uma user story de investigação - tempo limitado (máx 2-3 dias) para pesquisar, prototipar ou aprender algo necessário antes de poder estimar ou implementar.

🎯 Quando Usar Spikes

  • Tecnologia nova: "Nunca usámos GraphQL"
  • Estimativa impossível: "Não sabemos complexidade"
  • Decisão arquitetural: "MongoDB vs PostgreSQL?"
  • Performance unknown: "Aguenta 10k req/s?"
  • Feasibility: "É possível fazer X?"

📋 Exemplo de Spike

Spike: Investigar real-time sync mobile/web

Timebox: 2 dias

Objetivos:

  • Comparar WebSockets vs SSE
  • POC com Socket.io
  • Estimar custos infra

Entregável: Relatório + recomendação

⚠️ Erros Comuns e Como Evitar

Erro 1: Stories Técnicas

❌ "Como developer, quero refatorizar DB"

✅ "Como utilizador, quero respostas 50% mais rápidas"

Erro 2: Stories Gigantes

❌ "Sistema completo de pagamentos"

✅ Dividir em 10+ stories pequenas

Erro 3: Sem Critérios

❌ Story sem definição de "pronto"

✅ Given-When-Then específicos

Erro 4: Sem Valor

❌ "Quero ver gráficos" (Porquê?)

✅ "Para entender meus gastos"
