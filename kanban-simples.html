Kanban para Developers

O guia essencial para começar já a trabalhar com fluxo e eficiência

🚀 O que é o Kanban?

Kanban é um sistema visual de gestão de trabalho que te ajuda a visualizar o fluxo, limitar o trabalho em progresso e maximizar a eficiência. Nascido na Toyota, adaptado para o desenvolvimento de software.

Mindset & Princípios

🧠 A Mentalidade Kanban

Kanban não é apenas um board com post-its. É uma mudança de mentalidade sobre como abordamos o trabalho.

👁️

Visualizar o Trabalho

Torna visível todo o trabalho da equipa. Não há tarefas escondidas ou surpresas.

🎯

Limitar o WIP

Work In Progress limitado. Foca em terminar, não em começar.

🌊

Gerir o Fluxo

O objetivo é o trabalho fluir suavemente da esquerda para a direita.

📐

Políticas Explícitas

Regras claras sobre quando mover cartões entre colunas.

🔄

Feedback Loops

Reuniões regulares para melhorar continuamente o processo.

📈

Melhoria Contínua

Evoluir o processo baseado em métricas e experiência.

💡 Valores Fundamentais para a Equipa

  • Transparência: Todo o trabalho é visível para todos
  • Colaboração: Ajudar colegas bloqueados antes de pegar novas tarefas
  • Foco: Terminar antes de começar algo novo
  • Respeito pelos limites: WIP limits são acordos de equipa, não sugestões
  • Melhoria contínua: O processo evolui com a equipa

WIP Limits - A Regra de Ouro

🚦 Porque Limitar o Trabalho em Progresso?

A regra mais importante do Kanban: Cada pessoa trabalha numa coisa de cada vez!

❌ Errado

Multitasking

  • João tem 5 tarefas "Em Progresso"
  • Context switching constante
  • Nada acaba, tudo demora mais
  • Stress e bugs aumentam
  • Reuniões para sincronizar tudo
Resultado: 5 tarefas 20% completas após uma semana
✅ Correto

Foco

  • João trabalha numa tarefa até acabar
  • Foco total, sem interrupções
  • Entrega valor rapidamente
  • Qualidade do código aumenta
  • Feedback mais rápido
Resultado: 3 tarefas 100% completas e em produção 
// Políticas WIP da Equipa (exemplo)
const wipLimits = {
    todo: 5,           // Máximo 5 tarefas prontas para começar
    emProgresso: 3,    // Máximo 3 tarefas em desenvolvimento simultâneo
    codeReview: 2,     // Máximo 2 PRs em revisão
    testing: 2,        // Máximo 2 funcionalidades em teste
    
    // Regra individual
    porPessoa: 1      // CRÍTICO: 1 tarefa por developer!
};

// Quando o limite é atingido:
if (coluna.tarefas.length >= wipLimits[coluna.nome]) {
    // PARAR de puxar novas tarefas
    // AJUDAR colegas a desbloquear trabalho
    // NUNCA quebrar o limite sem consenso da equipa
}

Otimizar o Fluxo

🌊 O que é o Fluxo?

Fluxo é o movimento suave e contínuo do trabalho através do sistema. O objetivo é minimizar o tempo desde "ideia" até "valor entregue".

💡
Ideia

Requisito identificado

👨‍💻
Desenvolvimento

Código + Testes

🔍
Review

QA + Code Review

🚀
Deploy

Valor entregue!

🎯 Como Otimizar o Fluxo

1. Identificar Bottlenecks

Onde é que as tarefas ficam paradas? Normalmente: Code Review, QA, Waiting for Deploy

2. Reduzir o Tamanho das Tarefas

Tarefas pequenas fluem mais rápido. Divide user stories grandes em tarefas de 1-2 dias máximo.

3. Automatizar o que for Possível

CI/CD, testes automatizados, linters - tudo que remove fricção do processo.

4. Pull System

Não empurres trabalho para a próxima fase. A pessoa seguinte puxa quando está pronta.

// Métricas de Fluxo para Monitorizar
const metricas = {
    leadTime: "Tempo total desde TODO até DONE",
    cycleTime: "Tempo desde início do trabalho até DONE", 
    throughput: "Número de itens completos por semana",
    wip: "Trabalho em progresso em cada momento",
    
    // Indicadores de problemas:
    tarefasBlockeadas: "Itens parados > 2 dias",
    tarefasVoltaram: "Itens que voltaram para trás",
    
    // Objetivo
    fluxoContínuo: leadTime < 3 && !tarefasBlockeadas
};

Swimlanes - Organização Visual

🏊 O que são Swimlanes?

Swimlanes são filas horizontais que dividem o board para organizar diferentes tipos de trabalho, prioridades ou equipas. Ajudam a visualizar e gerir múltiplos fluxos de trabalho em paralelo.

Exemplo: Board com Swimlanes

🔥 Urgente
Bug crítico prod
Hotfix payment
🚀 Features
Nova dashboard
API v2
Integração Stripe
Sistema notificações
🐛 Bugs
Fix login mobile
Corrigir CSS IE11
🔧 Tech Debt
Refactor auth
Update deps
Migrar para TS

📐 Tipos Comuns de Swimlanes

  • Por Prioridade: Urgente / Normal / Nice-to-have
  • Por Tipo: Features / Bugs / Tech Debt / Experiments
  • Por Equipa: Frontend / Backend / DevOps / Mobile
  • Por Cliente: Cliente A / Cliente B / Interno
  • Por Serviço: API / Web App / Mobile App / Admin
  • Classes de Serviço: Expedite / Fixed Date / Standard / Intangible

💡 Boas Práticas com Swimlanes

  • Não exageres: 3-5 swimlanes são suficientes. Mais que isso confunde.
  • WIP limits por swimlane: Podes ter limites diferentes para cada fila.
  • Expedite lane: Uma fila especial para emergências (max 1 item, sempre).
  • Políticas claras: Define o que entra em cada swimlane.
  • Revisão regular: As swimlanes devem evoluir com as necessidades da equipa.

⚠️ Problemas Comuns e Soluções

🚫 "Mas eu preciso trabalhar em várias coisas!"

Solução: Não, não precisas. É uma ilusão. Experimenta 2 semanas com WIP=1 e vais ver a diferença. Se estás bloqueado, ajuda outros em vez de começar algo novo.

🚫 "O board não reflete a realidade"

Solução: Atualiza o board em tempo real, não no fim do dia. Cada movimento de tarefa deve ser imediato. Usa ferramentas digitais se a equipa é remota.

🚫 "Code Review demora dias"

Solução: Code review é trabalho prioritário! Antes de pegar nova tarefa, verifica se há PRs à espera. Limite de 24h para review.

🚫 "Tarefas muito grandes"

Solução: Nenhuma tarefa deve demorar mais de 3 dias. Divide em subtarefas. Se não consegues dividir, é porque não está bem definida.
