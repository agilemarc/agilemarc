O Que São User Stories?

Descrições simples de funcionalidades na perspetiva do utilizador

Formato Clássico:

Como [tipo de utilizador]
Quero [ação/funcionalidade]
Para que [benefício/valor]
Exemplo Bancário
Como cliente do banco
Quero consultar o meu saldo em tempo real
Para que possa gerir as minhas finanças eficazmente

📝 Características Essenciais

  • ✨ Simples: Linguagem não-técnica
  • 👤 Centradas no utilizador: Foco em quem usa
  • 💎 Orientadas ao valor: Sempre com propósito claro
  • 🤝 Negociáveis: Ponto de partida para conversas

Valor das User Stories para Equipas

Benefícios mesmo sem framework Scrum

🎯 Foco no Cliente

Mantém a equipa centrada nas necessidades reais dos utilizadores, não em requisitos técnicos abstratos.

Exemplo: Em vez de "Implementar API REST", temos "Cliente consulta extracto mensal"

💬 Comunicação Melhorada

Linguagem comum entre negócio e TI, reduzindo mal-entendidos e retrabalho.

Impacto: -40% em defeitos por má comunicação

🔄 Flexibilidade

Fácil repriorização baseada em valor de negócio, adaptando-se a mudanças do mercado.

Exemplo: COVID-19 → Priorizar funcionalidades digitais

📊 Estimação Simplificada

Histórias pequenas e comparáveis facilitam estimativas mais precisas.

Resultado: Previsibilidade +60% nas entregas
💡 Sem Scrum? User Stories funcionam em qualquer metodologia - Kanban, Waterfall híbrido, ou processos ad-hoc!

Anatomia de uma Boa User Story

Os 3 C's essenciais

📋 Card (Cartão)

A descrição escrita da história

Como gestor de conta
Quero ver alertas de transações suspeitas
Para proteger os meus clientes de fraude

💭 Conversation (Conversa)

Discussões que clarificam detalhes

  • • Que tipos de transações?
  • • Qual o threshold de suspeita?
  • • Como notificar o gestor?
  • • Tempo de resposta esperado?

✅ Confirmation (Confirmação)

Critérios de aceitação claros

Dado que existe transação >€5000 fora do país
Quando processada no sistema
Então alerta aparece em <30 segundos
⚠️ Erro Comum: Escrever romances! User Stories devem ser conversas, não contratos detalhados.

Critérios INVEST

Como avaliar a qualidade de uma User Story

I - Independent

Independente de outras histórias

N - Negotiable

Negociável em scope e solução

V - Valuable

Valor claro para o utilizador

E - Estimable

Estimável pela equipa

S - Small

Pequena (cabe num sprint)

T - Testable

Testável objetivamente

Exemplo INVEST - Banco

✅ Boa Story (INVEST completo):

"Como cliente, quero receber notificação SMS de transações >€100 para detetar uso não autorizado"

  • ✓ Independente: Não depende de outras funcionalidades
  • ✓ Negociável: Valor do threshold pode ser discutido
  • ✓ Valiosa: Protege cliente de fraude
  • ✓ Estimável: Equipa conhece SMS gateway
  • ✓ Small: 3-5 dias de trabalho
  • ✓ Testável: Transação €101 → SMS enviado

Como Escrever User Stories Eficazes

Processo passo-a-passo

1️⃣ Identificar o Utilizador

Quem vai beneficiar desta funcionalidade?

Personas Bancárias: Cliente Particular | Cliente Empresa | Gestor de Conta | Auditor | Compliance Officer

2️⃣ Definir a Necessidade

O que o utilizador precisa fazer?

Dica: Use verbos de ação - consultar, transferir, aprovar, cancelar, configurar

3️⃣ Clarificar o Valor

Porque é importante? Qual o benefício?

Valores Comuns: Poupar tempo | Reduzir erros | Aumentar segurança | Melhorar experiência

4️⃣ Adicionar Critérios de Aceitação

Como sabemos que está completo?

Formato Given-When-Then:
Dado que [contexto]
Quando [ação]
Então [resultado esperado]

Exemplos Práticos - Contexto Bancário

User Stories reais para diferentes áreas

💳 Pagamentos

Como cliente empresa
Quero agendar pagamentos em lote aos fornecedores
Para otimizar o processo de tesouraria

Critérios:
  • Upload ficheiro CSV com até 1000 pagamentos
  • Validação IBAN em tempo real
  • Aprovação em 2 níveis para valores >€50.000

🔒 Segurança

Como cliente particular
Quero definir limites diários para transferências
Para proteger a minha conta de fraude

Critérios:
  • Limites configuráveis por tipo de operação
  • Notificação quando limite atingido
  • Override com autenticação forte

📊 Compliance

Como compliance officer
Quero gerar relatórios de transações suspeitas automaticamente
Para cumprir regulamentos AML/KYC

Critérios:
  • Deteção baseada em regras configuráveis
  • Exportação em formato regulamentar
  • Audit trail completo

💰 Crédito

Como cliente particular
Quero simular crédito habitação com diferentes cenários
Para tomar decisão informada sobre financiamento

Critérios:
  • Variação de prazo, entrada e taxa
  • Cálculo TAEG em tempo real
  • Guardar simulações para comparação

Story Mapping - Visão Geral

Organizando histórias para criar produtos coerentes

O que é Story Mapping?

Técnica visual para organizar user stories numa estrutura bidimensional que representa a jornada do utilizador

🗺️ Eixo Horizontal

Fluxo temporal ou sequência de atividades do utilizador

📊 Eixo Vertical

Prioridade e sofisticação das funcionalidades

✨ Benefícios do Story Mapping

  • ✓ Visão holística do produto
  • ✓ Identificação de gaps funcionais
  • ✓ Priorização visual e intuitiva
  • ✓ Alinhamento equipa-stakeholders
  • ✓ Planeamento de releases incremental
💡 Dica: Story Mapping é especialmente útil para produtos novos ou grandes refactorings!

Como Criar um Story Map

Processo passo-a-passo

1️⃣ Identificar Atividades (Backbone)

Grandes blocos de atividade que o utilizador realiza

Exemplo Banco: Abrir Conta → Configurar Conta → Usar Conta → Gerir Conta

2️⃣ Decompor em Tarefas

Quebrar atividades em tarefas específicas do utilizador

"Usar Conta": Consultar Saldo | Transferir | Pagar | Receber

3️⃣ Adicionar User Stories

Colocar stories específicas sob cada tarefa

"Transferir":
• Transferência nacional
• Transferência SEPA
• Transferência internacional

4️⃣ Priorizar Verticalmente

Organizar por releases - MVP no topo

Release 1: Funcionalidades essenciais
Release 2: Melhorias UX
Release 3: Features avançadas

Técnicas de Splitting

Como dividir stories grandes (épicos) em stories INVEST

📱 Por Interface

Web → Mobile → API

Original: Sistema de alertas
Split: Alertas email | SMS | Push

⚡ Por Performance

Funcional → Rápido → Otimizado

Original: Relatório instantâneo
Split: <5min | <30seg | Real-time

🔢 Por Dados

Tipo A → Tipo B → Tipo C

Original: Todos pagamentos
Split: SEPA | Internacional | Crypto

📊 Por Regras Negócio

Simples → Complexo → Edge cases

Original: Aprovação crédito
Split: <10K auto | <50K manual | >50K comité

🎯 Por Operação

Create → Read → Update → Delete

Original: Gestão beneficiários
Split: Adicionar | Listar | Editar | Remover

👥 Por Papel

Cliente → Gestor → Admin

Original: Dashboard completo
Split: Vista cliente | Vista gestor | Vista compliance

Tasks, Spikes e Breakdown Técnico

Como decompor User Stories em trabalho técnico

🎯 Da User Story ao Trabalho Técnico

User Story: "Como cliente, quero autenticação biométrica para aceder mais rapidamente"

📋 Tasks Técnicas Derivadas:
  • Implementar SDK biometria iOS
  • Implementar SDK biometria Android
  • Criar serviço backend para token biométrico
  • Atualizar esquema BD para guardar hash biométrico
  • Implementar fallback para PIN
  • Criar testes unitários e integração
  • Atualizar documentação segurança

📝 Tasks

O que são: Trabalho técnico específico para implementar uma User Story

Características:
  • • Duração: 1-8 horas
  • • Atribuível a 1 pessoa
  • • Resultado verificável
  • • Não entrega valor isoladamente
Exemplo Banco:
"Configurar certificado SSL para API pagamentos"

🔬 Spikes

O que são: Investigação time-boxed para reduzir incerteza técnica

Quando usar:
  • • Tecnologia desconhecida
  • • Avaliar alternativas
  • • Estimar trabalho complexo
  • • Prova de conceito
Exemplo Banco:
"Investigar integração com novo provider KYC (max 3 dias)"

🐛 Bugs/Defeitos

O que são: Correções de problemas encontrados

Gestão:
  • • Priorizar por impacto
  • • Podem gerar tasks
  • • Tracking separado
  • • SLA por severidade
Exemplo Banco:
"Fix: Saldo incorreto após transferência cancelada"

💼 Technical Debt

O que é: Trabalho técnico adiado que precisa ser endereçado

Tipos:
  • • Refactoring
  • • Upgrade dependências
  • • Melhorias performance
  • • Documentação técnica
Exemplo Banco:
"Migrar de Java 8 para Java 17 no serviço pagamentos"

📊 Processo de Breakdown

User Story
Valor de Negócio
→
Sprint Planning
Discussão Técnica
→
Tasks
1-8 horas cada
⚠️ Regra de Ouro: User Stories descrevem O QUÊ e PORQUÊ (negócio). Tasks descrevem COMO (técnico). Mantenha esta separação clara!
💡 Best Practices:
  • ✓ Spikes sempre time-boxed (máx 2-3 dias)
  • ✓ Tasks devem ser completadas em 1 dia idealmente
  • ✓ Technical debt: alocar 15-20% da capacidade
  • ✓ Criar tasks durante Sprint Planning, não antes
  • ✓ Uma Story pode ter 3-10 tasks tipicamente

💡 Exemplos Práticos - Banco

XS (1 ponto, 2-4h)

"Adicionar tooltip ajuda no formulário"

• Mudança UI simples
• Sem lógica backend
• Testing mínimo

M (5 pontos, 2 dias)

"Implementar autenticação 2FA por SMS"

• Frontend + Backend
• Integração SMS gateway
• Testes segurança

L (8 pontos, 3-4 dias)

"Dashboard analytics para gestor conta"

• Múltiplos gráficos
• Queries complexas
• Performance tuning

XL (13 pontos, 5-6 dias)

"Sistema aprovação crédito workflow"

• Múltiplos estados
• Regras negócio complexas
• Integração sistemas
⚠️ Importante: Nunca converta diretamente! Use ranges e considere incerteza. Uma story de 5 pontos pode levar 1-3 dias dependendo de fatores externos.

Erros Comuns e Como Evitá-los

Aprender com a experiência

❌ Erro 1: Stories Técnicas

Errado: "Como developer, quero refatorizar a base de dados"

Correto: "Como cliente, quero respostas 50% mais rápidas nas consultas"

❌ Erro 2: Stories Gigantes

Errado: "Como utilizador, quero um sistema completo de homebanking"

Correto: Dividir em dezenas de stories pequenas e independentes

❌ Erro 3: Sem Critérios de Aceitação

Errado: Story sem definição clara de "pronto"

Correto: Critérios específicos, mensuráveis e testáveis

❌ Erro 4: Ignorar o "Para quê"

Errado: "Como cliente, quero ver gráficos" (Porquê?)

Correto: "Como cliente, quero ver gráficos para entender os meus padrões de gasto"

💡 Dica Pro: Faça retrospetivas sobre stories mal escritas - é a melhor forma de melhorar!

Conclusão - Pontos Essenciais

O que levar desta apresentação

🎯 Takeaways Principais

  • User Stories são sobre conversas, não documentação pesada
  • Foco sempre no valor para o utilizador, não em tarefas técnicas
  • Critérios INVEST garantem qualidade e são essenciais
  • Story Mapping cria visão holística e alinhamento
  • Estimação é sobre relatividade, não precisão absoluta
  • Funciona em qualquer metodologia, não apenas Scrum
  • Refinamento contínuo é chave para sucesso

🚀 Lembre-se Sempre

"As melhores User Stories nascem de conversas,
não de documentos."

User Stories bem escritas = Produtos de sucesso = Clientes satisfeitos
