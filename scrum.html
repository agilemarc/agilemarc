O Que é Scrum?

Framework leve para gerar valor através de soluções adaptativas

📖 Definição Oficial (Scrum Guide 2020)

"Scrum é um framework leve que ajuda pessoas, equipas e organizações a gerar valor através de soluções adaptativas para problemas complexos."

🧠 Teoria Empírica

Conhecimento vem da experiência e tomada de decisões baseadas no que é observado

Banco: Ajustar features baseado em uso real dos clientes

🔄 Iterativo e Incremental

Entregas frequentes de valor para obter feedback rápido

Banco: Lançar MVP app mobile, iterar baseado em feedback

🎯 Foco em Valor

Maximizar valor entregue ao cliente e negócio

Banco: Priorizar features que aumentam adoção digital
⚠️ Importante: Scrum NÃO é uma metodologia! É um framework - fornece estrutura mas não prescreve como fazer o trabalho técnico.
✅ Quando Usar Scrum:
  • • Problemas complexos com requisitos emergentes
  • • Necessidade de feedback frequente
  • • Equipas multidisciplinares de 3-9 pessoas
  • • Ambiente que suporta mudança

Como os Valores se Aplicam no Contexto Bancário

🦁 Coragem

Exemplo: Equipa tem coragem de parar desenvolvimento de feature complexa quando descobre que não traz valor esperado ao cliente, mesmo após 3 sprints investidos.

🎯 Foco

Exemplo: Durante Sprint para implementar Strong Customer Authentication (SCA), equipa recusa pedidos urgentes de outras features para manter foco no objetivo regulatório.

🤝 Compromisso

Exemplo: Equipa compromete-se com Sprint Goal de "Lançar pagamentos instantâneos" e todos colaboram para ultrapassar bloqueios técnicos com sistema legacy.

🙏 Respeito

Exemplo: Developers respeitam conhecimento do PO sobre regulação bancária; PO respeita estimativas técnicas sobre integração com core banking.

📖 Abertura

Exemplo: Equipa é transparente sobre riscos de segurança descobertos, mesmo que atrase lançamento da feature de open banking.

Os 3 Pilares Empíricos do Scrum

Base teórica do controlo de processos empíricos

🔍 Transparência

O processo emergente e o trabalho devem ser visíveis para quem executa e recebe o trabalho.

Prática Bancária:
  • Dashboard com métricas de Sprint visível a todos
  • Definition of Done clara para compliance
  • Backlog público com prioridades de negócio

🔎 Inspeção

Os artefactos Scrum e o progresso devem ser inspecionados frequentemente para detetar variações.

Prática Bancária:
  • Daily Scrum para verificar progresso SCA
  • Sprint Review com stakeholders compliance
  • Métricas de adoção digital revistas

🔄 Adaptação

Se aspetos do processo desviam dos limites aceitáveis, o processo deve ser ajustado.

Prática Bancária:
  • Ajustar Sprint length baseado em releases regulatórias
  • Adaptar DoD após auditoria segurança
  • Mudar composição equipa por skills gap
🔄 Eventos Formais de Inspeção e Adaptação:
  • • Sprint Planning: Inspecciona Product Backlog, adapta Sprint Backlog
  • • Daily Scrum: Inspecciona progresso, adapta plano diário
  • • Sprint Review: Inspecciona Increment, adapta Product Backlog
  • • Sprint Retrospective: Inspecciona processo, adapta forma de trabalhar

A Scrum Team

Unidade fundamental de Scrum - pequena e coesa

Características da Scrum Team:
  • ✓ Cross-functional: todas as competências necessárias
  • ✓ Self-managing: decide quem faz o quê, quando e como
  • ✓ 10 ou menos pessoas (recomendado)
  • ✓ Sem sub-equipas ou hierarquias
  • ✓ Responsável por todas as atividades relacionadas com o produto

📦 Product Owner

1 pessoa

  • • Maximizar valor do produto
  • • Gerir Product Backlog
  • • Decisões de produto

🎯 Scrum Master

1 pessoa

  • • Eficácia da Scrum Team
  • • Facilitar eventos
  • • Remover impedimentos

💻 Developers

3-9 pessoas

  • • Criar Increment utilizável
  • • Qualidade técnica
  • • Adaptar plano diário
Exemplo Equipa Banco
Projeto: Nova App Mobile Banking
  • Product Owner: Maria Silva (Background: Digital Banking Manager)
  • Scrum Master: João Santos (Certificado PSM, ex-developer)
  • Developers:
    • 2 Backend Engineers (Core banking integration)
    • 2 Mobile Developers (iOS/Android)
    • 1 UX/UI Designer
    • 1 QA Engineer (Especialista pagamentos)
    • 1 Security Engineer (Compliance PSD2)

À Scrum Team

📅 Definição: Sprints são eventos de duração fixa de um mês ou menos para criar consistência. Um novo Sprint começa imediatamente após o fim do anterior.

Sprint Planning

Inicia o Sprint estabelecendo o trabalho a ser realizado

⏱️ Timebox: Máximo 8 horas para Sprint de 1 mês (proporcional para Sprints menores)
👥 Participantes: Toda a Scrum Team (pode convidar outros para aconselhar)

📝 3 Tópicos do Sprint Planning

Tópico 1: Porquê este Sprint é valioso?

PO propõe como o produto pode aumentar valor. Toda a equipa colabora para definir Sprint Goal.

Exemplo: "Aumentar adoção digital oferecendo pagamentos P2P simples"

Tópico 2: O que pode ser Done neste Sprint?

Developers selecionam items do Product Backlog. Discussão com PO para entender melhor.

Exemplo: 5 User Stories de pagamento P2P + 2 bugs críticos

Tópico 3: Como o trabalho será realizado?

Developers planeiam trabalho para transformar items em Increment que atende DoD.

Exemplo: Decomposição em tasks técnicas, identificação de dependências
Sprint Planning - Banco

Agenda Real - 4 horas (Sprint 2 semanas)

09:00-09:30 Review contexto e métricas
PO apresenta feedback clientes, métricas adoção, prioridades negócio
09:30-10:30 Definir Sprint Goal
Discussão e agreement no objetivo: "Lançar P2P com telemóvel"
10:30-11:30 Selecionar Backlog Items
Velocity: 35 pontos. Selecionam 32 pontos + buffer
11:30-13:00 Decomposição e planeamento

Daily Scrum

Inspeção do progresso em direção ao Sprint Goal

⏱️ Timebox: 15 minutos SEMPRE
📅 Quando: Todos os dias, mesma hora e local
👥 Participantes: Developers (PO e SM apenas se estão trabalhando em items)

🎯 Propósito

  • Inspecionar progresso para Sprint Goal
  • Adaptar Sprint Backlog conforme necessário
  • Ajustar plano de trabalho do dia
  • Identificar impedimentos

📋 Estrutura

Developers escolhem formato. Exemplos:

  • 3 perguntas clássicas
  • Walk the board
  • Por funcionalidade
  • Round-robin

  • ❌ Status report para o gestor
  • ❌ Problem-solving meeting (discussões depois)
  • ❌ Única vez que a equipa fala no dia

Sprint Review

Inspeção do Increment e adaptação do Product Backlog

⏱️ Timebox: Máximo 4 horas para Sprint de 1 mês
👥 Participantes: Scrum Team + stakeholders convidados
📝 Output: Product Backlog atualizado

🎯 Atividades da Sprint Review

Apresentar

  • Increment funcional
  • O que foi Done
  • O que não foi Done
  • Sprint Goal achievement

Discutir

  • Feedback dos stakeholders
  • Mudanças no mercado
  • Novas oportunidades
  • Budget e timeline

Adaptar

  • Atualizar Product Backlog
  • Repriorizar items
  • Adicionar novos items
  • Remover obsoletos

Planear formas de aumentar qualidade e eficácia

⏱️ Timebox: Máximo 3 horas para Sprint de 1 mês
👥 Participantes: Apenas Scrum Team (ambiente seguro)
📝 Output: Ações de melhoria para próximo Sprint

Pessoas

Interações, comunicação, colaboração, conflitos

Processos

Cerimónias, práticas, workflow, Definition of Done

Ferramentas

Tecnologia, ambientes, automação, CI/CD

Product Backlog

Lista emergente e ordenada do que é necessário para melhorar o produto

📝 Commitment: Product Goal
👤 Owner: Product Owner (responsabilidade única)
🔄 Natureza: Vivo, nunca completo, constantemente refinado

📋 Características do Product Backlog

Ordenado

Items no topo são mais valiosos, detalhados e prontos

Emergente

Evolui com aprendizagem sobre produto, utilizadores e mercado

Estimado

Items têm estimativas de esforço (responsabilidade Developers)

Refinado

Atividade contínua de adicionar detalhes e estimativas

Increment & Definition of Done

Stepping stone concreto em direção ao Product Goal

📦 Increment: Soma de todos os Product Backlog items completados durante um Sprint + incrementos anteriores
✅ Definition of Done: Descrição formal do estado do Increment quando atende as medidas de qualidade requeridas

📦 Increment

  • Deve ser utilizável
  • Múltiplos por Sprint permitidos
  • Entregue na Sprint Review
  • Não precisa de "release" para ser Increment
Exemplo: Feature P2P completa, testada e deployable (mesmo que não released)

✅ Definition of Done

  • Cria transparência
  • Todos entendem igual
  • Se não atende DoD, não pode ser released
  • Commitment do Increment
Evolução: DoD deve ficar mais rigoroso com maturidade da equipa
Definition of Done - Banking App

DoD para User Story - Equipa Mobile Banking

  • ✅ Código
    • • Code review por 2 developers
    • • Sem warnings do linter
    • • Segue coding standards banco
  • ✅ Testing
    • • Unit tests >80% coverage
    • • Integration tests pass
    • • QA manual em staging
    • • Security scan sem vulnerabilidades críticas
  • ✅ Documentação
    • • API documentada (Swagger)
    • • README atualizado
    • • Release notes escritas
  • ✅ Compliance
    • • GDPR compliance verificada
    • • PSD2 requirements met
    • • Audit log implementado
  • ✅ Deploy
    • • Deployable para produção
    • • Feature flag configurada
    • • Rollback plan documentado

Anti-Padrões Scrum Comuns

Erros frequentes e como evitá-los

❌ "Scrum-But"

Anti-padrão: "Fazemos Scrum, MAS não precisamos de retrospectives"

Problema: Sem melhoria contínua, equipa estagna

Solução: Implementar TODOS os elementos de Scrum primeiro, adaptar depois

❌ "Proxy Product Owner"

Anti-padrão: PO sem autoridade real, precisa aprovals para tudo

Problema: Decisões lentas, equipa bloqueada

Solução: Empoderar PO ou encontrar alguém com autoridade real

❌ "Mini-Waterfall Sprints"

Anti-padrão: Análise → Design → Dev → Test em fases separadas

Problema: Sem colaboração, descoberta tardia de problemas

Solução: Trabalho paralelo, colaboração constante, vertical slicing

❌ "Scrum Master Gestor"

Anti-padrão: SM atribui tarefas e reporta status à gestão

Problema: Equipa não desenvolve auto-organização

Solução: SM facilita, equipa decide; transparência via artefactos

✅ Sinais de Scrum Saudável

  • ✓ Sprints previsíveis e sustentáveis
  • ✓ Equipa empoderada e motivada
  • ✓ Stakeholders satisfeitos e envolvidos
  • ✓ Melhoria mensurável sprint a sprint
  • ✓ PO disponível e decisor
  • ✓ SM remove impedimentos eficazmente
  • ✓ Developers colaboram constantemente
  • ✓ Qualidade nunca comprometida
